# Review: orca

**URL:** https://github.com/stablyai/orca
**Reviewed:** 2026-07-15

## Заявлено (Claimed)

Orca — десктоп-приложение (Electron, macOS / Windows / Linux) под лозунгом
**«The AI Orchestrator for 100x builders»**. Обещание: запускать несколько кодинг-агентов
(**Codex, Claude Code, OpenCode, Pi**) параллельно, **каждого в своём git-worktree**, и следить
за всеми из одного окна. Плюс мобильный компаньон (iOS/Android) для контроля агентов с телефона,
«автоматизации», встроенные skill-guides и подключение к личным API-подпискам (не по токенам, а по
подписке — в интерфейсе есть счётчики расхода лимита). Ставится как готовый бинарник с onorca.dev
(есть и Homebrew Cask, и APK). На момент ревью: **19 469 ★, +5 724 за неделю**, версия `1.4.142-rc.6`.

## Работает (Works)

Прогон вёлся из исходников в песочнице (`episodes/2026-07-ep001/sandbox/orca`), клон `--depth 1`.

- **Установка зависимостей проходит.** `pnpm install` (в изоляции — см. «Сломалось») отработал за
  **51.2 с**. `postinstall` честно пересобирает нативные модули под ABI Electron: `node-pty`
  собран из исходников, `cpu-features` прогнан через `node-gyp`, бинарь Electron 43.1.0 доустановлен.
  В логе — `[rebuild] Patched node-pty build artifacts are missing; rebuilding from source.`, без
  единой ошибки (`ELIFECYCLE`/`gyp ERR` в логе нет).
- **Сборка и запуск работают.** `pnpm dev` собрал main-процесс (бандл `out/main/index.js` ~7.1 МБ),
  preload и renderer-dev-server на `localhost:5173`, затем поднял Electron:
  `electron main process built successfully` → `starting electron app...` →
  `DevTools listening on ws://127.0.0.1:9382/…`.
- **Это настоящий многопроцессный десктоп-апп.** После старта в системе живут: main-процесс Orca,
  GPU-helper, network-service, renderer и **фоновый daemon** (`daemon-entry.js` со своим unix-сокетом
  и токеном). Приложение создало каталог состояния `~/Library/Application Support/orca-dev/` с
  подпапками `daemon/`, `cli/`, `codex-runtime-home/`, `agent-hooks/`, `blob_storage/` — т.е. под
  оркестрацию Codex действительно заложена инфраструктура.
- **Первый запуск отдаёт вылизанный онбординг.** Через CDP снят реальный UI (скриншот:
  `assets/first-run.png`): тёмная тема, сайдбар `Onboarding checklist / Tasks / Automations /
  Orca Mobile`, пустое состояние `No workspaces found` → `Add a project to get started` с кнопками
  **Add Project / Create workspace**, подсказки хоткеев (⌘N и т.д.) и внизу живые счётчики лимита
  **«85% used 5h · 63% used wk»** — ровно та «личная подписка», о которой пишет README.
- **Продукт большой и реальный.** 8 594 файла в репо, **7 033** файла исходников `.ts/.tsx`,
  8 встроенных агентских skill-ов (`orchestration`, `computer-use`, `orca-cli`, `orca-linear`…).
  Последний коммит — `fix(terminal): … (#8842)`: номер PR под 8 800 говорит о живом
  высокочастотном проекте, а не о пустышке.

## Сломалось (Broke)

- **`pnpm install` сначала попытался снести `node_modules` РОДИТЕЛЬСКОГО workspace.** Песочница
  вложена в проект `yt-production`, у которого в корне есть `pnpm-workspace.yaml`; pnpm поднялся вверх
  по дереву и счёл orca частью нашего workspace. Первый запуск упал с
  `ERR_PNPM_ABORTED_REMOVE_MODULES_DIR_NO_TTY  Aborted removal of modules directory due to no TTY` —
  и это нас спасло: без TTY он **отказался** удалять корневой `node_modules`. Проверено: корневой
  `node_modules` и `pnpm-lock.yaml` остались нетронуты. Лечится изоляцией:
  `pnpm -C … install --ignore-workspace`. **Это артефакт вложенности песочницы, а не баг Orca** —
  пользователь, клонирующий Orca отдельно, этого не увидит (важно для REPRO).
- **Frozen-lockfile mismatch.** Под `CI=true` pnpm по умолчанию ставит из «замороженного» lock-файла
  и упал: `ERR_PNPM_LOCKFILE_CONFIG_MISMATCH … "patchedDependencies" configuration doesn't match the
  value found in the lockfile`. Обошёл `--no-frozen-lockfile`. На чистой машине обычный `pnpm install`
  (без `CI`) в frozen-режим не уходит — пользователь, вероятно, тоже не упрётся, но проверить вживую.
- **Требуется Node 24, на машине — Node 22.22.1.** `engines: {"node":"24"}`; pnpm выдаёт
  `WARN Unsupported engine: wanted {"node":"24"} (current v22.22.1)`. **Предупреждение, не блокер** —
  установка/сборка/запуск прошли на Node 22. Но это реальная точка трения для юзера на LTS-ветке.
- **Оркестрацию по существу проверить не удалось.** Реальная суть продукта — гонять Codex/Claude
  Code/Pi параллельно — требует установленных агентских CLI и API-ключей. По правилам безопасности
  песочницы ключи в неё не передаются, поэтому дальше пустого онбординга (Add a project) прогон не
  шёл. Дошли до рабочего first-run, но не до живой мультиагентной сессии.

## Подозрительно (Suspicious)

- **Safety-gate:** один авто-исполняемый хук на install — `postinstall: node
  config/scripts/rebuild-native-deps.mjs`. Прочитан целиком: это подробно закомментированная
  пересборка нативных модулей под Electron (`@electron/rebuild` → `node-gyp`), без обращений в сеть
  сверх обычного реестра. Плюс `prepare: husky` (git-хуки). Пайп-в-шелл (`curl|sh`, `wget|bash`,
  `iwr|iex`), base64/`eval`-блобы, левые сетевые вызовы в build-скриптах — **не найдены**.
- **Положительный сигнал по цепочке поставок:** собственный `.npmrc` Orca ставит
  `minimum-release-age=4320` — 3-суточный карантин на свежие релизы зависимостей. Проект сам
  заботится о supply-chain, это скорее «за», чем «против».
- **Звёздная форензика (грубо, без глубокого замера):** признаков накрутки не видно.
  8 800+ PR, 7 000+ файлов исходников, вылизанный многопроцессный продукт, релизы под бинарники,
  мобильное приложение — масштаб инженерии соответствует масштабу звёзд. Всплеск +5.7k/нед похож на
  органический хайп вокруг рабочего продукта, а не на манипуляцию. (Полноценную star-forensics —
  velocity vs contributors/age/issues — здесь не проводил; при желании добить на этапе скрипта.)
- **Мелочь-настораживающая:** на first-run счётчики лимита уже показывают расход
  («85% used 5h»). На абсолютно чистой машине это должно быть по нулям — проверить на записи, не
  подтягивает ли dev-сборка лимиты из уже установленной прод-версии Orca (на машине есть и
  `~/Library/Application Support/orca`, и `orca-dev`).

  Electron».
- Момент старта: **один `pnpm dev` — и в системе взрывается пачка процессов** (main, GPU, renderer,
  network-service) **плюс фоновый daemon** со своим сокетом. Хук: «запустил одно окно — получил
  демона в фоне».

  `Add a project to get started`, и внизу уже тикающие счётчики **«85% used 5h · 63% used wk»**.
  Хук: «лимит подписки капает ещё до того, как ты добавил хоть один проект».

  главное обещание («100x, агенты параллельно») в песочнице не проверяемо. Хук: «чтобы увидеть те
  самые 100x, придётся отдать ему свои ключи».

## Черновой вердикт (Draft verdict lean)

**ГОДНОТА (с оговоркой)** — по существу это НЕ накрутка и НЕ «рано»: перед нами зрелый, вылизанный,
реально работающий продукт (v1.4, 7 000+ файлов исходников, PR под #8842, готовые бинарники и
мобильный клиент), который ставится, собирается и запускается без грязных трюков, а
safety-gate чист (единственный `postinstall` — честная пересборка нативы, да ещё и `.npmrc` с
supply-chain-карантином). Масштаб инженерии соответствует масштабу звёзд, всплеск +5.7k/нед выглядит
органическим. Единственное, что удерживает от чистой ГОДНОТЫ и держит открытым риск **ХАЙП**, — само
ядро обещания («100x builders», параллельные агенты) в песочнице непроверяемо без API-ключей и CLI
агентов, а маркетинговые суперлативы всегда идут впереди реальности. **Что должен сделать хост, чтобы
закрыть вопрос:** записать живую сессию — добавить проект, поднять 2–3 агента в параллельных
worktree и показать, реально ли это ускоряет работу или просто красиво выглядит. Если ускоряет —
твёрдая ГОДНОТА; если это дорогой оркестратор ради оркестратора — понижаем до ХАЙП.
